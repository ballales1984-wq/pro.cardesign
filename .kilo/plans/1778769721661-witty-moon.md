## 2. Bug Critici

### 2.1 `t.py` вЂ” File Python corrotto
- **File:** `t.py`
- **Problema:** Le istruzioni `import` sono spezzate su righe separate senza logica. Il file ГЁ sintatticamente non valido.

- **Impatto:** Crash immediato all'esecuzione. Probabile file di appunti incompleto.

### 2.2 Bug CSS: `--surface` non definito
- **File:** `styles/main.css`, riga 102-105
- **Problema:** La regola `.tool.active` usa `var(--surface)` ma la variabile CSS definita ГЁ `--bg-surface` (riga 10). Risultato: il pulsante dello strumento attivo non mostra alcuno sfondo evidenziato.
```css

  background: var(--surface);  /* в†ђ non esiste, dovrebbe essere var(--bg-surface) */

  border-color: var(--accent2);
}
```

### 2.3 Backend Python completamente non funzionante
- **File:** `backend_api.py`, `simple_server.py`, `run_server.py`
- **Problema:** Tutti i file Python importano da `voxel_editor`, `voxel_scaling`, `pyramid_decimate` вЂ” moduli **inesistenti** nel repository. Nessuna route Flask ГЁ raggiungibile.
- **Impatto:** Il backend ГЁ inutile. L'app Electron non comunica con il backend Python (preload.js espone solo `saveFile`/`loadFile` via dialog file system, non API HTTP).

## 3. Problemi di StabilitГ 

### 3.1 Performance вЂ” Rendering senza culling
- **File:** `src/voxel-engine.js`
- **Problema:** Ogni voxel ГЁ un `THREE.Mesh` separato aggiunto alla scena. Non c'ГЁ frustum culling, occlusion culling, nГ© batching. Con 1000+ voxel la scena degrada pesantemente.
- **Evidenza:** La raycast in `_raycast()` itera su `this.meshes.values()` senza strutture spaziali.

### 3.2 Memory вЂ” Shared Geometry disposal
- **File:** `src/voxel-engine.js`, `clearAll()` (riga 322-341)
- **Problema:** `dispose()` della geometria condivisa e dei materiali nella cache, ma dopo il clear si crea una **nuova** geometria. I vecchi mesh referenziati in `this.meshes` non vengono rimossisi dal memory se non dalla scena.

### 3.3 Undo/Redo potenziale memory leak
- **File:** `src/voxel-engine.js`, `_pushHistory()` e `redo()`
- **Problema:** `_pushHistory` limita a 50 voci, ma `_redoStack` non ha limite. Inoltre, `_addVoxelSilently` chiamato dal redo crea nuovi oggetti THREE.Mesh senza mai rimuovere quelli vecchi se il key ГЁ duplicato.

### 3.4 Nessuna validazione input in `fromJSON()`
- **File:** `src/voxel-engine.js`, riga 426-443
- **Problema:** `fromJSON()` non valida la struttura dei dati. Un JSON malformato o corrotto puГІ crashare l'intera app.

## 4. Miglioramenti Possibili

### 4.1 рџ”ґ Architettura вЂ” Unificare frontend/backend
- Il backend Flask ГЁ completamente scollegato dall'app Electron. Decidere se:
  - **(a)** Rimuovere il backend Python e fare tutto nel renderer Three.js, oppure
  - **(b)** Collegarlo via IPC/HTTP effettivo (aggiungere `axios` o `node-fetch` nel main process).
- I moduli Python (`mesh_utils.py`, `voxel_to_stl.py`, `physics_sim.py`) sono duplicati logicamente rispetto ai JS equivalenti.

### 4.2 рџ”ґ Performance вЂ” InstancedMesh
- Sostituire i singoli `THREE.Mesh` per ogni voxel con `THREE.InstancedMesh`. Questo ridurrebbe le draw call da N (numero voxel) a 1 per materiale. Miglioramento potenziale: **10-50x** con scene grandi.

### 4.3 рџџЎ Performance вЂ” Spatial indexing per raycast
- Implementare un **octree** o **hash grid** per la raycast. Attualmente l'intera lista di mesh viene testata. Con `InstancedMesh` il raycast cambierГ  approccio (usare `raycast` nativo di Three.js su istanze).

### 4.4 рџџЎ Gestione risorse Three.js
- Aggiungere `renderer.dispose()`, rimuovere correttamente listener eventi, e pulire le texture al cambio scena o chiusura finestra.

### 4.5 рџџЎ Marching Cubes effettivo
- `mesh-exporter.js` riconosce che `_marchingCubes` ГЁ un placeholder. Implementarlo (o usare una libreria come `mcubes` o `dual-contouring`) per esportazioni mesh di qualitГ  superiore.

### 4.6 рџџЎ Aggiornare il piano di sviluppo
- `DEVELOPMENT_PLAN.md` menziona pygame ed editor 2.5D, ma il progetto usa Electron/Three.js. Risanare la documentazione.

### 4.7 рџџў Aggiungere test
- Nessun test unitario. Aggiungere almeno test per:
  - `MaterialSystem` (aggiunta/rimozione materiali)
  - `ModuleSystem` (albero gerarchico, JSON serialization)
  - `PhysicsCalc` (massa, CoM, inerzia con casi noti)
  - `VoxelEngine` (add/remove, undo/redo, JSON round-trip)

### 4.8 рџџў Correggere `.gitignore`
- `.gitignore` esclude `*.json` ma il progetto ha `full_model.json` e `test_model.json` (tra i file modificati/untracked). Questi file non verranno mai tracciati.

### 4.9 рџџў Transparenza materiale
- `glass` ha `transparent: true, opacity: 0.6` nelle proprietГ  materiale, ma il `MeshStandardMaterial` in `_getMaterial()` non applica mai queste proprietГ .

### 4.10 рџџў Fill layer migliorato
- `fillLayer()` attualmente crea solo il perimetro di un rettangolo 5x5. Dovrebbe offrire l'opzione di riempimento solido.
